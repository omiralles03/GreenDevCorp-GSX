# Users, Groups & Access Control (Week 4)

This weeks objectives were:

1. **User & Group Structure:** Organize users and groups to reflect the team's roles and enforce least privilege.
2. **File Permissions:** Allow shared work without exposing private data using advanced access control (Unix permissions, SGID, Sticky bit).
3. **Resource Limits per User:** Prevent one user from exhausting system resources using PAM (Pluggable Authentication Modules).
4. **Environment Personalization:** Give each user a customized shell environment without manual setup.
5. **Security Reasoning:** Verify that the setup actually enforces least privilege using automated scripts.

## Project Architecture

1. **Dynamic User & Group Administration (`add_users_group.sh`)**:
   * A highly dynamic script that accepts any user prefix, user count, and group name as parameters (e.g., `./add_users_group.sh dev 4 greendevcorp`).
   * It intelligently parses `/etc/passwd` to find the highest existing ID for the given prefix and auto-increments from there. This ensures no existing users are ever overridden and allows seamless team scaling.
   * Generates the base team directory at `/home/[group_name]`.

2. **Advanced Access Control (`setup_secpecific_configs.sh`)**:
   * Applies a strict least privilege model using specific permission bits:
   * **`/home/greendevcorp/bin` (0750)**: Owned by `root:greendevcorp`. The team group can execute scripts (`r-x`), but cannot modify or delete them. Others have no access (`---`).
   * **`/home/greendevcorp/shared` (3770)**: Owned by `root:greendevcorp`. Implements **SGID (2)** so all new files inherit the `greendevcorp` group, and the **Sticky Bit (1)** so users can only delete their own files. The team has full `rwx` access, while others are blocked (`---`).
   * **`/home/greendevcorp/done.log` (0644)**: Owned by `dev1:greendevcorp`. `dev1` is the only authorized user to write/append tasks to the log (`rw-`), while the rest of the team can only read it (`r--`), preventing forged entries.

3. **Resource Limits**:
   * Applied via PAM through `/etc/security/limits.d/greendevcorp.conf`.
   * It hard-limits the resources for anyone in the `@greendevcorp` group: `nproc` (max processes: 700), `nofile` (open files: 4096), `rss` (memory: ~2GB), and `cpu` (max time: 15 mins).

4. **Environment Personalization**:
   * A global profile drop-in (`/etc/profile.d/greendevcorp_env.sh`) is injected.
   * Whenever a user of the `greendevcorp` group logs in, they automatically inherit the shared `bin` directory in their `$PATH`, custom aliases (`shared`, `dlog`), and a personalized green bash prompt (`PS1`).

5. **Security Verification (`verification_w4.sh`)**:
   * An automated script that tests the permission structures (using `stat`) and checks the hard limits (`ulimit -H`) to guarantee compliance with the least privilege model.

## Design Decisions

* **SGID + Sticky Bit over ACLs**: We decided to combine the SGID and the Sticky Bit for the `shared` folder instead of complex POSIX ACLs. SGID ensures smooth collaboration by forcing a common group on new files, and the Sticky bit provides the exact protection needed against accidental deletions.
* **Drop-in Profile (`/etc/profile.d/`)**: Instead of appending configurations to each user's `~/.bashrc` manually, we used the `/etc/profile.d/` directory. This ensures that the environment is centralized: if we want to add a new alias for the whole team, we only need to change one file.
* **PAM for User Limits**: We used PAM `limits.conf` instead of systemd `cgroups` for these specific restrictions because we needed to limit the resources generated by *human users/sessions* rather than limiting a background daemon/service.

## Hints and Questions to Guide Your Thinking

* **If a file is in a shared directory and owned by dev1, but needs to be readable by all team members, what permissions would you set? Why?**
  
  We would set `0640` (`rw-r-----`) and ensure the file belongs to the `greendevcorp` group. This grants read/write access to the owner (`dev1`), read-only access to the team members in the group, and zero access to external users, enforcing the principle of least privilege.

* **What's the difference between setgid on a directory vs. a file? Why would you use each?**
  
  * **On a directory (SGID):** Any new file created inside the directory automatically inherits the directory's group ownership, rather than the primary group of the user who created it. We use this in our `shared` folder so the team can effortlessly edit each other's files.
  * **On an executable file (SGID):** The program executes with the privileges of the file's group, rather than the privileges of the user running it. We would use this if a command needs to access files owned by a specific group, without giving the user direct access to those files.

* **If you misconfigure permissions and a user can't access a file they need, how do you troubleshoot?**
  
  First, I would use the `id <username>` command to verify if the user is actually part of the required group. Then, I would use `ls -la` or `stat` to check the ownership and permissions of the file. If that is correct, the issue might be the parent directories: a user needs execute (`x`) permissions on every parent directory to traverse the path to the file. 

* **How would you verify that your permission model actually enforces the security policy you intended?**
  
  The best way is to test it empirically. You switch to a non-privileged user (e.g., `su - dev2`) and actively try to break the rules: try to delete a file owned by `dev1` in the `shared` directory (should fail due to the Sticky Bit), or try to write to `done.log` (should fail due to lack of write permissions). Our script `verification_w4.sh` automates this verification.
